Этот документ является **единственным каноническим источником истины** для проектирования и поэтапной реализации омниканального sales-бота на базе LLM.

Все дальнейшие артефакты — структура репозитория, workflow-файлы n8n, контракты, prompt-файлы, ingestion-пайплайны, адаптеры к внешним системам, тестовые сценарии и инструкции для Codex — должны:

1. выводиться из этой мастер-спецификации;
2. не противоречить этой мастер-спецификации;
3. при расхождении трактоваться в пользу этой мастер-спецификации.

Если в процессе реализации выясняется, что какое-либо решение нельзя корректно вывести из текущего текста, это считается не поводом для импровизации, а поводом для:

- фиксации пробела;
- уточнения спецификации;
- внесения новой версии документа.

Импровизация, скрытая логика и “достройка по ощущению” запрещены.

## 2. Цель продукта

### 2.1. Бизнес-цель

Создать полноценного омниканального чат-бота на базе LLM, который в перспективе заменит классические lead-capture / chat-конструкторы уровня Leadback и будет выполнять функцию первичного sales-менеджера компании.

Система должна:

- принимать входящие обращения из цифровых каналов;
- вести диалог в консультативном sales-стиле;
- выявлять потребность клиента;
- предлагать релевантные услуги, решения и направления компании;
- не выдумывать факты о продуктах и услугах;
- при необходимости корректно передавать диалог оператору;
- автоматически собирать и создавать лиды;
- быть расширяемой на новые каналы без перестройки ядра.

### 2.2. Продуктовая цель

Собрать модульную систему, в которой:

- **Chatwoot** выступает операторским и каналовым входом;
- **n8n** выступает оркестратором интеграций и состояний на уровне workflow;
- **LLM** вызывается через HTTP;
- **RAG** использует знания сайта и связанных материалов через Qdrant;
- **lead creation** производится сначала в промежуточный ingest/endpoint, затем в CRM;
- **handoff** в оператора является штатной частью процесса, а не аварийным исключением.

### 2.3. Цель текущей мастер-спеки

Дать локальному агенту Codex достаточно строгую, подробную и машиночитаемо структурированную постановку, чтобы он мог:

1. создать правильную структуру проекта;
2. последовательно собрать решение по этапам;
3. не уйти в псевдо-no-code компромиссы;
4. не спрятать бизнес-логику в Code node, expressions или неявные хаки;
5. выдать проверяемый, тестируемый, развиваемый baseline.

## 3. Методологические принципы

### 3.1. Spec-Driven Development

Проект реализуется строго по схеме:

1. фиксируются цели, границы и инварианты;
2. фиксируются данные и контракты;
3. фиксируются сценарии и workflow;
4. фиксируются критерии приёмки;
5. только после этого создаётся реализация.

### 3.2. Принцип 0-1-2-3

Каждая итерация разработки должна проходить через четыре обязательные стадии:

0. **Исследовать** — понять задачу, контекст, ограничения, причины, способ решения;
1. **Исправить / Реализовать** — внести изменения в workflow, конфигурации, код, контракты;
2. **Протестировать** — выполнить проверки, сценарии, тест-кейсы, эмуляцию потока;
3. **Отчитаться** — зафиксировать что сделано, чем подтверждено, какие риски остались.

### 3.3. Принцип читаемой оркестрации

Основная бизнес-логика продукта должна быть читаема на уровне диаграммы и структуры workflow. Нельзя допускать, чтобы критическая логика была спрятана:

- в Code node;
- в длинных expressions, заменяющих программный код;
- в неочевидных raw JSON-конфигурациях;
- в скрытых побочных эффектах внешних сервисов.

### 3.4. Принцип безопасной неопределённости

Если система не имеет достаточных оснований для точного ответа, она обязана:

- не выдумывать;
- либо дать ограниченный честный ответ с оговоркой;
- либо перевести диалог в режим handoff;
- либо запросить уточнение в допустимых пределах.

### 3.5. Принцип канонического восстановления

Удалённый прежний репозиторий не имеет статуса истины. Любые уцелевшие обрывки старых артефактов считаются только историческим сырьём. Каноническая система должна выводиться из текущего документа.

### 3.6. Принцип контролируемого исполнения через Codex и subagents

Subagents могут использоваться как слой исполнения локального Codex, но не считаются частью runtime-архитектуры продукта.

Правила:

- продуктовая спецификация описывает бот и его интеграционный контур, а не внутреннюю организацию агентной работы Codex;
- parent-agent остаётся ответственным за итоговую интерпретацию master-spec, интеграцию результатов и финальный отчёт;
- subagents целесообразны прежде всего для read-heavy exploration, documentation verification, contract mapping, evidence collection и reviewer-pass;
- parallel write-heavy исполнение не считается режимом по умолчанию и требует явного контроля;
- отсутствие subagents не должно ломать применимость этой спецификации: пакет должен оставаться рабочим и в single-agent режиме.

## 4. Область охвата

### 4.1. Что входит в первую полную baseline-спецификацию

В baseline входят следующие слои:

1. вход сообщений через Chatwoot;
2. оркестрация через n8n;
3. dedup и anti-loop guardrails;
4. диалоговый слой с LLM без RAG;
5. handoff на оператора;
6. snapshot-ingestion сайта;
7. RAG retrieval через Qdrant;
8. sales state machine;
9. сбор полей лида;
10. запись лида в промежуточный target API;
11. подготовка адаптера к CRM;
12. наблюдаемость и pilot-readiness;
13. расширение на новые каналы через Chatwoot inbox model.

### 4.2. Что не входит в baseline первого мастер-документа

Ниже перечислены явные non-goals, которые не должны размывать первую реализацию:

- тонкая оптимизация prompt engineering по десяткам вариантов;
- полноценный intent-classifier на отдельной ML-модели;
- сложный multi-agent runtime внутри самого продукта;
- автоматическое принятие коммерчески значимых решений без операторского контроля;
- сложный CRM-bi-directional sync на первом проходе;
- генерация файлов, КП, медиаконтента или персонализированных смет без отдельной спецификации;
- полноценная аналитическая DWH-архитектура;
- мультиязычная пользовательская логика в baseline, кроме будущей расширяемости.

## 5. Системный контур

### 5.1. Высокоуровневая схема

Базовый контур системы:

`Канал -> Chatwoot -> Webhook -> n8n -> Guardrails -> Context/RAG -> LLM -> Decision -> Chatwoot reply / Handoff / Lead Target API / Internal Note`

### 5.2. Роли компонентов

#### Chatwoot

Назначение:

- единая входная точка сообщений;
- хранение conversation thread;
- рабочее место оператора;
- каналовый шлюз;
- место публикации ответа бота;
- место фиксации internal note и статусов диалога.

#### n8n

Назначение:

- оркестрация событий и бизнес-переходов;
- дедупликация и anti-loop;
- вызовы LLM через HTTP;
- вызовы retrieval / Qdrant / snapshot-пайплайнов;
- маршрутизация handoff;
- сбор слотов лида;
- отправка данных в Lead Target API;
- логирование и трассировка.

#### LLM Endpoint

Назначение:

- генерация ответа на основе system-policy, истории, retrieval-контекста и текущего состояния диалога;
- выделение структурированных сигналов, если это предусмотрено отдельным prompt-контрактом.

#### Qdrant

Назначение:

- векторное хранилище знаний;
- поиск релевантных чанков по пользовательскому запросу;
- опора для grounded-ответов.

#### Lead Target API

Назначение:

- первый стабильный write-контур для создания лида;
- точка идемпотентного приёма lead payload;
- адаптационный слой между ботом и будущей CRM.

#### CRM Adapter

Назначение:

- последующая интеграция с выбранной CRM;
- не должен ломать остальной контур при смене CRM.

## 6. Архитектурные инварианты

Ниже перечислены правила, нарушение которых считается архитектурным дефектом.

### 6.1. Инвариант единой входной точки

Все каналы должны заходить через Chatwoot как через операторский и событийный фасад. Новые каналы не должны изменять центральную бизнес-логику, а должны подключаться через channel-specific inbox внутри Chatwoot.

### 6.2. Инвариант оркестрации

Вся бизнес-оркестрация находится в n8n workflow-слое. Не допускается переносить центральную маршрутизацию в скрытые промежуточные сервисы без отдельной ревизии мастер-спеки.

### 6.3. Инвариант запрета Code node

Используются только стандартные ноды n8n. `Code`, `Function`, `Function Item` и любые их эквиваленты запрещены.

### 6.4. Инвариант запрета псевдо-no-code обходов

Запрещается переносить сложную логику:

- в гигантские expressions;
- в длинные JavaScript-вставки в параметрах нод;
- в raw-конфиги, фактически заменяющие код;
- в нечитабельные inline-преобразования без отдельного явного шага workflow.

Если логика не выражается прозрачно стандартными нодами, это надо либо:

- декомпозировать на workflow-шаги;
- вынести в отдельный явно специфицированный сервис;
- пересмотреть постановку.

### 6.5. Инвариант grounded-ответа

Система не должна выдавать факты о продуктах и услугах, не подтверждённые знаниями сайта, approved knowledge sources или явно заданными policy-правилами.

### 6.6. Инвариант честного ограничения

При низкой уверенности или слабом retrieval бот не должен компенсировать пробел фантазией.

### 6.7. Инвариант handoff-as-feature

Передача оператору — штатная функциональность, а не отказ системы.

### 6.8. Инвариант lead idempotency

Повторные сообщения, повторная отправка webhook и повторные попытки записи не должны плодить дубли лидов в окне идемпотентности.

### 6.9. Инвариант канальной расширяемости

Подключение новых каналов не должно требовать переписывания sales-логики, RAG-логики и lead capture логики.

### 6.10. Инвариант проверяемости

Любой этап реализации должен доказываться тест-кейсами, логами, sample payload и acceptance-результатами.

## 7. Пользовательские и операторские роли

### 7.1. Конечный пользователь

Пишет компании в web-chat, а в перспективе в Telegram и другие каналы, ожидая:

- быстрый ответ;
- понятное объяснение услуг и решений;
- помощь в выборе;
- возможность оставить контакты и получить продолжение;
- возможность перейти к человеку.

### 7.2. Оператор

Работает в Chatwoot и ожидает:

- прозрачную историю диалога;
- понятный признак handoff;
- краткое summary от бота;
- уже собранные поля лида;
- отсутствие зацикливания и дублей;
- отсутствие мусорных “почти одинаковых” лидов.

### 7.3. Администратор / интегратор

Ожидает:

- прозрачные workflow;
- понятную наблюдаемость;
- управляемую переиндексацию знаний;
- безопасное расширение каналов;
- переносимость интеграции CRM.

## 8. Канонический glossary

- **Conversation** — единица диалога в Chatwoot.
- **Inbound message** — сообщение, пришедшее от пользователя в канал.
- **Outbound bot reply** — сообщение, которое публикует бот в Chatwoot.
- **Internal note** — служебная запись для оператора внутри диалога.
- **Handoff** — перевод управления диалогом оператору.
- **Lead** — структурированная запись о потенциальном клиенте.
- **Lead Target API** — промежуточный write-контур перед CRM.
- **Knowledge snapshot** — воспроизводимый слепок сайта/источника знаний на момент времени.
- **Knowledge chunk** — минимальный поисковый фрагмент в RAG.
- **Grounded answer** — ответ, опирающийся на знания, а не на выдумывание.
- **Dedup key** — ключ, позволяющий не реагировать дважды на одно и то же событие.
- **State store** — хранилище краткоживущего состояния workflow.
- **Correlation ID** — сквозной идентификатор обработки одного входного события.

## 9. Продуктовые требования

### 9.1. Основные пользовательские возможности

Система должна уметь:

1. принимать входящие сообщения из Chatwoot;
2. не отвечать дважды на один и тот же inbound event;
3. не реагировать на собственные bot/operator сообщения;
4. поддерживать диалог в sales-манере;
5. выявлять потребность;
6. задавать ограниченное число уточняющих вопросов;
7. презентовать решения компании без выдумывания;
8. опираться на RAG-контекст после подключения knowledge layer;
9. корректно сообщать об ограничениях знания;
10. инициировать handoff при соответствующих условиях;
11. собирать минимальный lead payload;
12. создавать lead record без дублей;
13. сообщать пользователю о следующем шаге;
14. формировать internal note для оператора.

### 9.2. Минимальный комплект данных лида

Lead должен поддерживать следующие поля:

- `external_user_id` — идентификатор пользователя в канале / Chatwoot contact / иной стабильный ключ;
- `conversation_id`;
- `channel`;
- `name` — если получено;
- `contact` — phone / email / username / handle;
- `contact_type`;
- `interest` — услуга / продукт / направление интереса;
- `description` — краткое описание запроса;
- `source` — конкретный канал / inbox / campaign; поле должно быть заполнено из channel/inbox/campaign metadata, даже если пользователь его не вводил напрямую;
- `status` — внутреннее processing/state поле lead-слоя или поле ответа Target API, но не обязательная часть исходного `Lead Payload`;
- `created_at`;
- `last_updated_at`;
- `dedup_fingerprint`.

### 9.3. Функциональные ограничения

Система не должна:

- обещать несуществующие услуги;
- придумывать цены, сроки, гарантии, кейсы и интеграции;
- создавать лид без минимально полезного payload, если нет fallback-правил;
- отвечать бесконечно длинными сообщениями;
- зацикливаться на повторных уточнениях;
- спорить с пользователем вместо эскалации;
- пытаться закрыть сложную ситуацию без handoff, если уверенность низкая.

## 10. Нефункциональные требования

### 10.1. Прозрачность

Все ключевые переходы системы должны быть объяснимы по логам и workflow.

### 10.2. Надёжность

Повторная доставка webhook-событий не должна ломать систему и плодить ответы.

### 10.3. Идемпотентность

Повторные записи lead payload не должны порождать множественные лиды в пределах окна идемпотентности.

### 10.4. Наблюдаемость

Должны существовать correlation-aware логи, позволяющие восстановить путь:

`message -> dedup -> state -> retrieval -> LLM -> decision -> reply/handoff/lead`

### 10.5. Расширяемость

Добавление нового канала не должно менять продуктовые правила sales-диалога.

### 10.6. Воспроизводимость знаний

Каждая knowledge-база должна быть привязана к versioned snapshot, а не к неявному “текущему состоянию сайта”.

### 10.7. Управляемость ошибок

Ошибки интеграций должны переводиться в предсказуемое поведение: retry, fallback, handoff, запись в логи.

---

## 11. Канальная стратегия

### 11.1. Порядок подключения каналов

Базовый порядок:

1. web chat через Chatwoot;
2. Telegram через Chatwoot inbox;
3. дальнейшие каналы по приоритету компании: WhatsApp, email, MAX или иные.

### 11.2. Правило расширения каналов

Подключение нового канала должно проходить через проверку только следующих слоёв:

- delivery of inbound event into Chatwoot;
- корректность channel metadata;
- соответствие reply-механизма Chatwoot;
- тесты дедупа и anti-loop;
- совместимость контактных слотов.

Логика sales, RAG, handoff и lead capture не должна переписываться из-за нового канала.

## 12. Диалоговая политика sales-бота

### 12.1. Роль бота

Бот — первичный sales-консультант компании. Он не изображает техподдержку, не играет в свободного собеседника и не пытается заменить человека в тех случаях, где требуется handoff.

### 12.2. Базовые задачи в диалоге

Бот должен:

1. приветствовать и мягко принимать запрос;
2. понять потребность;
3. уточнить контекст в разумных пределах;
4. сопоставить запрос с услугами/решениями;
5. дать grounded-объяснение;
6. предложить следующий шаг;
7. собрать lead payload;
8. при необходимости передать оператору.

### 12.3. RU-only

Во всех пользовательских ответах baseline-бот отвечает на русском языке. Исключения должны быть отдельно специфицированы позднее.

### 12.4. Ограничение числа уточнений

Бот не должен задавать бесконечные вопросы. В baseline вводится ограничение:

- не более 2–3 уточняющих шагов подряд без перехода к ценности, предложению, CTA или handoff.

### 12.5. Поведенческий тон

Тон должен быть:

- деловой;
- вежливый;
- не бюрократический;
- не агрессивный;
- не избыточно маркетинговый;
- не фамильярный;
- не многословный без пользы.

### 12.6. Запрет на выдумывание

Если в knowledge source нет подтверждения услуги, функции, кейса, цены или процесса, бот не должен утверждать это как факт.

### 12.7. Следующий шаг

Каждый содержательный ответ sales-бота должен вести разговор вперёд: уточнение, предложение, CTA, контакт, handoff.

---

## 13. State machine диалога

В baseline вводится ограниченная machine-state модель на уровне n8n/state store.

### 13.1. Список состояний

- `intro`
- `qualify`
- `propose`
- `cta`
- `lead_capture`
- `handoff`
- `closed`

### 13.2. Назначение состояний

#### `intro`
Первичный вход, приветствие и приём первого запроса.

#### `qualify`
Уточнение задачи, контекста, типа услуги, масштаба, проблемы.

#### `propose`
Предложение релевантного решения, направления или набора услуг.

#### `cta`
Подведение к следующему шагу: оставить контакт, получить консультацию, подключить специалиста.

#### `lead_capture`
Сбор минимально необходимого payload для лида.

#### `handoff`
Передача на оператора.

#### `closed`
Диалог завершён, либо лид создан и дальнейший шаг зафиксирован, либо разговор передан оператору.

### 13.3. Правила переходов

- `intro -> qualify`
- `qualify -> propose`
- `propose -> cta`
- `cta -> lead_capture`
- `lead_capture -> closed`
- любое состояние -> `handoff`, если выполнены условия эскалации
- `handoff -> closed`, если управление передано оператору

### 13.4. Запреты

- нельзя вечно удерживать диалог в `qualify`;
- нельзя создавать лид до появления минимально полезного набора данных, если нет fallback-исключения;
- нельзя возвращать разговор из `handoff` обратно боту без отдельной политики.

---

## 14. Условия handoff

### 14.1. Handoff обязателен в случаях

1. пользователь явно просит человека / менеджера / оператора;
2. бот не имеет достаточной уверенности или retrieval пуст / слаб;
3. запрос затрагивает коммерчески чувствительные детали без подтверждённых знаний;
4. бот не смог собрать контакт после ограниченного числа попыток;
5. пользователь недоволен, раздражён или просит нестандартное решение;
6. требуется действие вне полномочий бота.

### 14.2. Действия при handoff

Система должна:

1. прекратить попытки автономно “додавить” ситуацию;
2. сообщить пользователю, что запрос передан специалисту;
3. создать internal note в Chatwoot;
4. приложить краткое summary;
5. передать уже собранные поля лида и контекста;
6. изменить статус / метку / признак разговора так, чтобы оператор видел handoff.

### 14.3. Минимум для handoff summary

Summary должен включать:

- кто обратился;
- о чём запрос;
- что уже выяснено;
- какие данные контакта есть;
- почему произошёл handoff;
- что ожидает пользователь.

## 15. Политика lead capture

### 15.1. Цель lead capture

Не просто “поймать контакт”, а собрать минимально полезную запись, которая позволит оператору и CRM-контруру продолжить работу без потери смысла разговора.

### 15.2. Обязательные слоты baseline

- `contact`
- `name` — если доступно
- `interest`
- `description`

### 15.3. Допустимые виды контакта

- телефон;
- email;
- Telegram username / иной ник, если он является реальным каналом обратной связи;
- channel-native identifier, если это разрешено политикой компании и достаточно для обратного контакта.

### 15.4. Минимальная валидация контакта

В baseline требуется:

- для email — базовая синтаксическая проверка;
- для телефона — нормализация и базовая проверка допустимого вида;
- для username/handle — отсутствие пустого или явно мусорного значения.

### 15.5. Повторный запрос контакта

Если контакт не получен, бот может запросить его повторно ограниченное число раз.

Baseline-правило:

- не более 2 повторных запросов контакта;
- затем handoff либо завершение по policy.

### 15.6. Поведение при частично собранном lead

Если интерес и описание уже собраны, но контакт не получен, система не должна молча терять контекст. Требуется зафиксировать partial lead context и передать его оператору при handoff.

## 16. Политика работы с CRM и Lead Target API

### 16.1. Базовое решение

Первая боевая запись должна происходить не напрямую в произвольную CRM, а в **Lead Target API** — отдельный стабильный целевой интерфейс.

### 16.2. Причины такого решения

Это даёт:

- стабильность контракта;
- независимость от выбора CRM;
- возможность идемпотентности и дедупа до CRM;
- контролируемый retry/failure handling;
- переносимость решения.

### 16.3. Требования к Lead Target API

Lead Target API должен поддерживать:

- идемпотентный приём payload;
- возврат `lead_id` или `existing_lead_id`;
- явный статус `created / duplicate / rejected / failed`;
- диагностическое сообщение ошибки;
- временную устойчивость к повторным запросам.

### 16.4. Дедуп lead creation

Дедуп строится минимум по сочетанию:

- контакт;
- интерес / тип услуги;
- временное окно.

Конкретный fingerprint может включать:

- нормализованный контакт;
- canonical interest;
- channel/source;
- bucket по времени.

### 16.5. Поведение после создания лида

После успешного создания лида система должна:

- уведомить пользователя о следующем шаге;
- записать internal note для оператора;
- сохранить `lead_id` в trace/log context;
- не создать второй лид по тому же событию.

## 17. Knowledge / RAG стратегия

### 17.1. Источник знаний

Первый источник знаний — контент сайта компании и согласованные с ним материалы, прошедшие ingestion.

### 17.2. Snapshot-first подход

Сайт не должен читаться как “живой хаос” прямо в момент ответа. Нужен воспроизводимый snapshot.

Минимальные стадии:

1. crawl / scrape сайта;
2. сохранение snapshot;
3. очистка и canonicalization;
4. chunking;
5. embeddings;
6. upsert в Qdrant;
7. version tagging.

### 17.3. Базовый инструмент snapshot-ingestion

В baseline допускается использование Firecrawl как инструмента обхода/съёма сайта. При этом конкретный инструмент не должен менять архитектурный принцип snapshot-first.

### 17.4. Требования к knowledge snapshot

Каждый snapshot должен иметь как минимум:

- `snapshot_id`;
- дату/время формирования;
- исходный домен;
- список захваченных URL;
- список исключённых URL;
- версию чанкинга;
- версию embedding model;
- статус загрузки в Qdrant.

### 17.5. Chunking policy

Chunking должен обеспечивать:

- смысловую целостность фрагмента;
- сохранение source URL;
- метаданные section/title;
- недопущение слишком больших нечитаемых блоков;
- пригодность для grounded-ответа.

### 17.6. Retrieval policy

В baseline требуется:

- dense retrieval по векторному поиску;
- top-k выборка;
- фильтрация пустых и явно слабых результатов;
- передача контекста в LLM только вместе с источниковыми метаданными.

### 17.7. Поведение при слабом retrieval

Если retrieval пуст или слаб, бот не должен симулировать знание. Допустимы только:

- ограниченный ответ с честной оговоркой;
- дополнительное уточнение;
- handoff.

### 17.8. Версионирование знаний

Все боевые ответы должны быть привязуемы к конкретной knowledge version, по крайней мере через trace/log.

## 18. Guardrails: дедупликация и защита от циклов

### 18.1. Цель

Гарантировать принцип:

**1 inbound event -> максимум 1 осмысленная реакция системы на этом уровне обработки**

### 18.2. Dedup key

Минимальный dedup key строится по:

- `conversation_id`
- `message_id`

Дополнительно может использоваться channel/inbox metadata.

### 18.3. TTL окна дедупа

Должно существовать краткоживущее хранилище dedup-records с TTL. Конкретное хранилище выбирается отдельной технической реализацией, но логика должна быть специфицирована независимо от него.

### 18.4. Anti-loop

Workflow обязан отбрасывать:

- сообщения, отправленные самим ботом;
- сообщения, отправленные оператором;
- системные события, не являющиеся новым пользовательским input.

### 18.5. Ошибка дедупа — критический дефект

Если система отвечает повторно на один и тот же inbound event, это считается критическим дефектом baseline-уровня.

### 19.1. Correlation ID

### 19.2. Минимальный набор логируемых этапов

- inbound received;
- normalized event built;
- dedup pass/fail;
- anti-loop pass/fail;
- conversation state loaded;
- retrieval invoked;
- retrieval result quality;
- LLM request sent;
- LLM response received;
- decision taken;
- Chatwoot reply published;
- handoff created;
- lead payload built;

### 19.3. Минимальные метрики baseline

- latency end-to-end;
- dedup suppress count;
- anti-loop suppress count;
- empty retrieval rate;
- handoff rate;
- lead creation rate;
- duplicate lead suppression rate;
- integration error rate;
- LLM timeout/error rate.

### 19.4. Pilot-readiness

До боевого частичного запуска должны быть видимы:

- доля handoff;
- доля лидов;
- случаи пустого retrieval;
- средняя задержка;
- доля ошибок публикации ответа.

## 20. Security и операционные ограничения

### 20.1. Секреты

Ключи LLM, Qdrant, Firecrawl, Chatwoot API и Lead Target API не хранятся в открытом виде в workflow-файлах и репозитории.

### 20.2. Логи

В логах нельзя без нужды раскрывать чувствительные персональные данные. Контактные данные должны логироваться в маскированном или ограниченном виде, если полные значения не требуются для операционной диагностики.

### 20.3. Отказоустойчивость

При недоступности отдельных интеграций система должна выбирать контролируемое поведение: retry, graceful failure, handoff, запись инцидента.

## 22. Workflow decomposition

Ниже описаны канонические workflow-единицы. При реализации они должны быть отдельными, читаемыми и тестируемыми. Один workflow не должен бесконтрольно поглощать всю логику проекта.

### 22.1. WF-01 — Inbound Echo Baseline

**Цель:** доказать, что Chatwoot -> n8n -> Chatwoot reply работает технически.

**Вход:** inbound webhook от Chatwoot.  
**Выход:** простой детерминированный ответ в Chatwoot.

**Требования:**

- корректно принять webhook;
- нормализовать минимальный event;
- опубликовать ответ;
- не выполнять сложную бизнес-логику;
- использоваться как базовый интеграционный smoke test.

### 22.2. WF-02 — Dedup and Anti-loop

**Цель:** гарантировать, что система не отвечает дважды и не реагирует на собственные сообщения.

**Вход:** normalized inbound event.  
**Выход:** либо пропуск в следующий workflow, либо suppress.

**Требования:**

- построить dedup key;
- проверить наличие записи в TTL store;
- создать dedup record при первом проходе;
- отфильтровать bot/operator/system messages;
- логировать suppress reason.

### 22.3. WF-03 — LLM Dialog Without RAG

**Цель:** получить стабильный RU-only диалоговый ответ без knowledge layer.

**Вход:** normalized event + conversation state.  
**Выход:** ответ, decision envelope, обновление state.

**Требования:**

- вызывать LLM через HTTP Request;
- задавать system-policy: sales role, RU-only, no hallucination;
- ограничивать длину ответа;
- обрабатывать timeout/retry;
- не придумывать grounding там, где его нет.

### 22.4. WF-04 — Handoff

**Цель:** технически корректно эскалировать разговор оператору.

**Вход:** decision envelope / explicit user request / fallback condition.  
**Выход:** user-facing reply + internal note + conversation flag/status.

**Требования:**

- фиксировать причину handoff;
- формировать summary;
- публиковать internal note;
- прекращать попытку самостоятельного закрытия кейса.

### 22.5. WF-05 — Site Snapshot Ingestion

**Цель:** получить воспроизводимый knowledge snapshot сайта.

**Вход:** домен / sitemap / ingestion trigger.  
**Выход:** snapshot dataset + Qdrant upsert result.

**Требования:**

- поддерживать repeatable run;
- version-tag snapshot;
- исключать мусорные/служебные страницы;
- строить canonical metadata;
- загружать знания в Qdrant.

### 22.6. WF-06 — RAG Answer Flow

**Цель:** подключить retrieval и grounded answer generation.

**Вход:** user message + state + knowledge version.  
**Выход:** grounded answer / handoff / limited reply.

**Требования:**

- выполнять retrieval top-k;
- оценивать силу retrieval;
- подмешивать контекст в LLM prompt;
- сохранять trace знания;
- не отвечать как будто знание есть, если его нет.

### 22.7. WF-07 — Sales State Machine

**Цель:** сделать поведение бота предсказуемым и стадийным.

**Вход:** conversation state + user message + LLM signals.  
**Выход:** next_stage + answer policy.

**Требования:**

- управлять стадиями `intro/qualify/propose/cta/lead_capture/handoff/closed`;
- ограничивать число уточнений;
- не допускать бесконечной квалификации;
- переводить в lead capture в правильный момент.

### 22.8. WF-08 — Lead Capture

**Цель:** стабильно собирать минимальный lead payload.

**Вход:** stage `lead_capture` + partial state.  
**Выход:** structured lead payload or handoff.

**Требования:**

- собирать `contact`, `name`, `interest`, `description`;
- валидировать contact;
- ограничивать число повторных запросов;
- сохранять partial progress.

### 22.9. WF-09 — Lead Write

**Цель:** создать lead в Lead Target API и не плодить дубли.

**Вход:** valid lead payload.  
**Выход:** write result + user reply + internal note.

**Требования:**

- строить dedup fingerprint;
- выполнять идемпотентную запись;
- корректно обрабатывать `duplicate` и `created`;
- сообщать пользователю следующий шаг;
- логировать `lead_id`.

### 22.10. WF-10 — Observability and Pilot Ops

**Цель:** обеспечить управляемый пилот и диагностику качества.

**Вход:** события всех workflow.  
**Выход:** trace/log/metric artefacts.

**Требования:**

- correlation-aware logging;
- понятные метрики;
- диагностика latency, handoff, empty retrieval, lead success/failure.

## 23. Требования к реализации n8n workflow

### 23.1. Общие требования

Каждый workflow должен:

- иметь явную цель;
- иметь описанный вход;
- иметь описанный выход;
- не скрывать критическую логику;
- быть пригодным для независимого тестирования.

### 23.2. Стандарты читаемости

Требуется:

- осмысленное имя workflow;
- осмысленные имена нод;
- комментарии/описания на уровне workflow, если это не вредит поддерживаемости;
- отсутствие неиспользуемых нод и параметров;
- явная группировка шагов по смыслу.

### 23.3. Допустимые способы ветвления

Ветвление должно строиться на стандартных нодах типа IF / Switch / Merge / HTTP Request / Set / Wait / Execute Workflow и аналогичных стандартных механизмах.

### 23.4. Недопустимые практики

Запрещено:

- упаковывать полсистемы в одну ноду;
- прятать бизнес-логику в expressions;
- дублировать почти одинаковые ветки без нужды;
- смешивать технические и продуктовые решения без явной границы;
- “быстро исправлять” поведение через хаки вместо корректного изменения спецификации.

## 24. Требования к prompt-слою

### 24.1. Разделение prompt-файлов

Prompt-слой должен быть вынесен в отдельные управляемые артефакты, а не захардкожен кусками внутри workflow, если его объём и важность это оправдывают.

Минимально должны существовать:

- system prompt для sales-bot;
- handoff summary prompt;
- lead extraction / structured decision prompt, если требуется;
- no-answer / groundedness policy.

### 24.2. Системные правила prompt-layer

Prompt обязан навязывать:

- RU-only;
- роль sales-консультанта;
- запрет на выдумывание фактов;
- обязательность handoff при нехватке оснований;
- ограничение многословности;
- движение диалога к следующему шагу.

### 24.3. Структурированные сигналы

Если LLM используется для принятия решения, желательно выдавать не только текст ответа, но и структурированные сигналы, пригодные для маршрутизации:

- `needs_handoff`
- `needs_contact`
- `confidence_band`
- `next_stage`
- `has_grounding`

## 26. План реализации по фазам

Codex должен реализовывать систему не “как получится”, а по зафиксированным волнам.

### Фаза 0 — Project Bootstrap

Результат:

- создан репозиторий;
- добавлена мастер-спека;
- созданы каталоги;
- созданы базовые contracts/prompts/workflow placeholders;
- оформлен AGENTS.md.

### Фаза 1 — Inbound Echo

Результат:

- работает Chatwoot -> n8n -> Chatwoot reply;
- есть smoke test сценарий;
- подтверждена publish capability.

### Фаза 2 — Dedup / Anti-loop

Результат:

- один inbound event не создаёт два ответа;
- bot/operator messages suppress-ятся;
- есть trace suppress reason.

### Фаза 3 — LLM Without RAG

Результат:

- бот отвечает на русском;
- соблюдает sales-role;
- не ломается при timeout/retry;
- не имитирует knowledge-grounding.

### Фаза 4 — Handoff

Результат:

- бот умеет корректно эскалировать;
- operator видит summary;
- пользователь получает корректное сообщение.

### Фаза 5 — Snapshot Ingestion

Результат:

- строится воспроизводимый knowledge snapshot;
- данные загружаются в Qdrant;
- фиксируется knowledge version.

### Фаза 6 — RAG Baseline

Результат:

- бот использует retrieval context;
- empty/weak retrieval обрабатывается безопасно;
- ответы становятся grounded.

### Фаза 7 — Sales State Machine

Результат:

- бот ведёт диалог стадийно;
- ограничено число уточнений;
- появляется контролируемый CTA.

### Фаза 8 — Lead Capture

Результат:

- бот стабильно собирает минимальный lead payload;
- контакт валидируется;
- partial lead не теряется.

### Фаза 9 — Lead Write

Результат:

- lead создаётся через Target API;
- duplicate suppression доказан;
- operator получает lead context.

### Фаза 10 — Observability / Pilot

Результат:

- внедрены логи и метрики;
- система пригодна к ограниченному пилоту;
- зафиксированы runbook и acceptance evidence.

## 27. Acceptance criteria

Ниже перечислены канонические критерии приёмки baseline. При реализации они должны быть декомпозированы в acceptance matrix и test scenarios.

### 27.1. Интеграционный контур

- входящее сообщение из Chatwoot доставляется в n8n;
- n8n может опубликовать ответ в Chatwoot;
- message flow воспроизводим на тестовом окружении.

### 27.2. Дедуп

- повторная доставка одного и того же сообщения не вызывает второй ответ;
- bot/operator/system messages не попадают в ответный контур;
- suppress события логируются.

### 27.3. LLM-слой

- ответы на пользовательском уровне на русском языке;
- бот не отвечает пусто без объяснимой причины;
- при сбое LLM не возникает неуправляемый провал без логов и fallback.

### 27.4. RAG

- при наличии релевантного knowledge snapshot retrieval возвращает usable context;
- бот использует контекст, а не игнорирует его;
- при empty/weak retrieval бот не начинает выдумывать.

### 27.5. Sales flow

- бот умеет переходить от приветствия к выявлению потребности;
- умеет предложить следующий шаг;
- не зависает в бесконечных уточнениях.

### 27.6. Handoff

- handoff можно вызвать явно по просьбе пользователя;
- handoff происходит при low-confidence / fallback-сценариях;
- operator получает summary.

### 27.7. Lead capture

- при наличии данных создаётся корректный lead payload;
- при отсутствии контакта бот делает ограниченное число повторных запросов;
- partial lead context сохраняется.

### 27.8. Lead write

- valid lead payload записывается в Lead Target API;
- дубли подавляются;
- успешная запись создаёт traceable lead_id.

### 27.9. Observability

- по одному `correlation_id` можно восстановить путь обработки;
- доступны ключевые метрики пилота.

### 27.10. Архитектурная дисциплина

- в workflow нет Code/Function node;
- нет скрытой бизнес-логики в expressions, заменяющих код;
- структура репозитория соответствует мастер-спеке.

## 28. Тестовая стратегия

### 28.1. Обязательные уровни проверок

1. **Smoke tests** — проверка базового прохода событий;
2. **Integration tests** — проверка взаимодействия компонентов;
3. **Acceptance scenarios** — проверка пользовательских и операторских флоу;
4. **Failure-path checks** — проверка timeout, empty retrieval, duplicate event, lead duplicate.

### 28.2. Канонические сценарии

Минимум должны быть покрыты сценарии:

- простой вход и ответ;
- повторная доставка одного webhook;
- сообщение от оператора;
- пользователь просит человека;
- retrieval strong;
- retrieval empty;
- успешный сбор контакта;
- неуспешный сбор контакта;
- создан новый лид;
- попытка создать дубликат лида.

## 29. Требования к AGENTS.md для локального Codex

AGENTS.md в репозитории должен закреплять поведение локального агента.

Минимально он должен требовать от Codex:

1. считать `master-spec.md` в корне репозитория каноническим источником истины;
2. не использовать Code node в n8n;
3. не прятать логику в expressions;
4. работать фазами, а не перепрыгивать через этапы;
5. обновлять контракты и тесты при изменении поведения;
6. фиксировать доказательства выполнения;
7. не придумывать поля и интеграции, не описанные в мастер-спеке;
8. при пробеле в спецификации помечать `SPEC_GAP`, а не импровизировать.

7. доказан lead write через Lead Target API;
8. доступны логи и ключевые метрики;
9. в workflow отсутствуют запрещённые Code/Function-узлы;
10. acceptance evidence собран и сохранён.

## 31. Риски и известные точки внимания

### 31.1. Главные архитектурные риски

- дрейф от n8n-orchestration к неявному middleware;
- вынос логики в Code node под предлогом “быстрее”;
- вынос логики в expressions под видом “формально без code node”;
- неявная зависимость от конкретной CRM;
- отсутствие устойчивого dedup store;
- плохое качество knowledge snapshot;
- слабый retrieval и выдумывающие ответы;
- неуправляемое разрастание prompt-слоя;
- отсутствие реальных acceptance-evidence.

### 31.2. Главные продуктовые риски

- бот слишком рано пытается “продавать”, не поняв задачу;
- бот слишком долго квалифицирует и раздражает пользователя;
- бот просит контакт слишком навязчиво;
- бот не умеет вовремя эскалировать;
- operator получает слишком мало контекста.

## 32. Решения, которые считаются зафиксированными

Считаются утверждёнными без дополнительного обсуждения, пока не появится новая версия мастер-спеки:

1. формат мастер-спеки — единый markdown-документ;
2. цель — спецификация, достаточная для локального Codex;
3. целевой охват — полный baseline по фазам 0–10;
4. n8n policy — только стандартные ноды, без Code node и псевдо-no-code обходов;
5. CRM strategy — через промежуточный Lead Target API;
6. удалённый прежний репозиторий не является источником истины.

## 33. Инструкции локальному Codex на старт реализации

Локальный Codex должен начать не с произвольного кода, а в следующем порядке:

1. создать репозиторий по канонической структуре;
2. положить этот документ в корень репозитория как `master-spec.md`;
3. создать schema placeholders в `contracts/`;
4. создать prompt placeholders в `prompts/`;
5. создать workflow placeholders по фазам;
6. оформить `AGENTS.md` с жёсткими правилами исполнения;
7. при необходимости добавить repo-local `.codex/` слой только для project-specific overrides и subagents, не смешивая его с продуктовой архитектурой;
8. реализовывать систему строго по фазам;
9. на каждом этапе добавлять test evidence и acceptance artefacts.

Codex запрещено:

- пропускать фазу дедупа ради “скорейшего MVP”;
- смешивать фазы без явной причины;
- создавать скрытые runtime-слои без отдельной фиксации в спецификации;
- менять архитектурные инварианты без новой версии документа.
