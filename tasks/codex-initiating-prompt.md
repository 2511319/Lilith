# Codex Initiating Prompt

Ниже дан стартовый prompt, который следует использовать как первый рабочий prompt для Codex в новом репозитории, созданном по этому spec-pack.

```text
Ты работаешь внутри репозитория омниканального LLM/RAG sales-бота.

Главная цель: реализовывать систему по spec-driven пакету документации, где `master-spec.md` в корне репозитория является каноническим source of truth.

Порядок истинности:
1. `master-spec.md`
2. `AGENTS.md`
3. `contracts/*.json`
4. тематические документы в `docs/`
5. задачи в `tasks/`

Работай строго по принципу 0-1-2-3:
0. Исследовать
1. Реализовать
2. Протестировать
3. Отчитаться

Жёсткие инварианты:
- не использовать `Code` / `Function` / `Function Item` и их аналоги в n8n (допустимы стандартные функциональные ноды библиотеки n8n);
- не прятать бизнес-логику в expressions, raw-конфигах и inline-JS;
- не перепрыгивать через фазы 0 → 10;
- не менять архитектурные инварианты без явной фиксации `SPEC_GAP`;
- не считать subagents частью runtime-продукта.

Контур продукта фиксирован:
Chatwoot → n8n → guardrails → retrieval/context → LLM → reply / handoff / Lead Target API / internal note.

CRM пишется не напрямую, а через Lead Target API.

Что сделать сейчас:
1. Прочитай `master-spec.md`, `AGENTS.md`, `docs/10-repo-skeleton.md`, `docs/11-contracts-overview.md`, `docs/12-implementation-plan.md`, `tasks/codex-bootstrap.md`, `tasks/phase-backlog.md`.
2. Сформируй краткий execution plan только для Фазы 0 — Bootstrap.
3. Отдельно оцени, нужны ли repo-local `.codex/` overrides или project-specific subagents сверх уже существующей глобальной конфигурации. Если не нужны — зафиксируй это и не создавай лишнего.
4. Выполни только Фазу 0 без перехода к Фазе 1.
5. После выполнения выдай отчёт в структуре:
   - что изучено;
   - что создано / изменено;
   - чем подтверждена корректность;
   - какие остались риски и `SPEC_GAP`.

Правило делегирования:
- если в окружении доступны subagents, используй их только там, где это повышает качество: read-heavy exploration, docs verification, contract mapping, evidence collection, reviewer-pass;
- parallel write-heavy исполнение не использовать как режим по умолчанию;
- финальная ответственность за интеграцию результатов всегда остаётся за родительским агентом.

Нельзя завершать фазу формально. Если артефакт создан только номинально, это не считается готовностью.
```

## Когда применять

- при первом запуске Codex в чистом репозитории, созданном по этому spec-pack;
- после переноса `master-spec.md`, `AGENTS.md`, `docs/`, `contracts/` и `tasks/` в новый repo;
- до начала Фазы 1.

## Что этот prompt не делает

- не заменяет `AGENTS.md`;
- не меняет product architecture;
- не требует обязательного использования subagents;
- не разрешает перепрыгивать через фазы.
