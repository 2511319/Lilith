# 05. Handoff Policy

## Принцип

Handoff — не отказ системы, а штатный механизм перевода разговора человеку.

## Триггеры handoff

### Явные
- пользователь просит человека;
- пользователь просит связаться менеджеру;
- пользователь отказывается продолжать диалог с ботом.

### Неявные
- low confidence;
- weak/empty retrieval в критичном вопросе;
- превышен лимит уточнений;
- бот не может собрать валидный контакт за ограниченное число попыток;
- интеграционная ошибка, не позволяющая продолжить нормальный поток.

## Что должно сделать решение

1. Поставить conversation в состояние `handoff`.
2. Сформировать `handoff summary`.
3. Записать internal note для оператора.
4. Отправить пользователю корректное сообщение о передаче.
5. Прекратить попытку “дожимать” кейс ботом.

## Требования к summary

Summary должен содержать:

- причину handoff;
- краткое описание запроса пользователя;
- уже собранные поля лида;
- рекомендуемое действие оператора.

## Запреты

- нельзя скрывать handoff под видом “я уточню и вернусь” без реальной передачи;
- нельзя после handoff возвращать диалог боту без отдельной политики;
- нельзя создавать handoff-summary без явной причины и минимального контекста.
