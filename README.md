# Omnichannel Sales Bot — Spec Pack

## Назначение

Этот архив — исходный пакет спецификаций для создания омниканального sales-бота на базе LLM.

Пакет предназначен для сценария, в котором:

- источником истины является не старый репозиторий и не устные договорённости, а формализованная спецификация;
- реализация выполняется локальным агентом Codex;
- архитектура строится по принципам spec-driven development;
- n8n используется только на стандартных нодах без `Code` / `Function`-логики и без псевдо-no-code обходов.

## Состав пакета

- `master-spec.md` — канонический master-spec и главный source of truth.
- `AGENTS.md` — жёсткие правила поведения локального Codex.
- `docs/` — тематические срезы мастер-спеки для удобной работы по слоям.
- `contracts/` — машинные JSON Schema-контракты.
- `tasks/` — исполнительные задания, initiating prompt и порядок старта реализации.

## Как использовать пакет

1. Создать новый чистый репозиторий.
2. Положить `master-spec.md` в корень репозитория как единственный canonical artifact.
3. Скопировать `AGENTS.md` в корень репозитория.
4. Перенести `contracts/`, `docs/` и `tasks/` без изменения смысла.
5. До начала фазовой реализации прочитать `tasks/codex-initiating-prompt.md` и использовать его как стартовый prompt для Codex.
6. Начинать реализацию только по фазам, зафиксированным в `master-spec.md` и `tasks/`.

## Что считается истинным

Приоритет истинности:

1. `master-spec.md`
2. `AGENTS.md`
3. `contracts/*.json`
4. тематические документы в `docs/`
5. задачи в `tasks/`

Если один файл противоречит master-spec, верным считается master-spec.  
Если в спецификации обнаружен пробел, реализация должна пометить его как `SPEC_GAP`, а не импровизировать.

## Каноническое направление проекта

- Вход сообщений: **Chatwoot**
- Оркестрация: **n8n**
- Вызов модели: **HTTP к LLM endpoint**
- Knowledge layer: **RAG через Qdrant**
- Первый write-контур: **Lead Target API**
- Эскалация: **handoff на оператора как штатная функция**
- Канальная стратегия: **web → Telegram → остальные каналы через Chatwoot inbox model**


## Важное уточнение по subagents

Subagents рассматриваются как слой исполнения Codex и контроля качества сборки. Они не являются частью runtime-архитектуры самого продукта и не меняют канонический продуктовый контур Chatwoot → n8n → LLM/RAG → Lead Target API / handoff.
