# AGENTS.md

## Статус файла

Этот файл обязателен для локального Codex и задаёт правила исполнения проекта.  
Он подчинён `master-spec.md` и не может ему противоречить.

## Главные правила

1. Считать `master-spec.md` каноническим источником истины.
2. Не использовать `Code`, `Function`, `Function Item` или любые их аналоги в n8n.
3. Не переносить сложную бизнес-логику в expressions, raw-конфиги и inline-JS.
4. Реализовывать систему строго по текущей канонической фазовой карте без скрытого перепрыгивания.
5. Любое изменение поведения сопровождать обновлением контрактов, тестов и evidence.
6. Не придумывать поля, интеграции, состояния, статусы и внешние зависимости, которых нет в спецификации.
7. При пробеле в спецификации помечать `SPEC_GAP` и останавливать произвольное расширение.
8. Любой спор между “быстрее” и “прозрачнее” решать в пользу прозрачности.


## Политика исполнения через subagents

Subagents разрешены только как вспомогательный слой исполнения Codex.
Они не являются частью runtime-продукта и не меняют каноническую архитектуру бота.

Разрешённые сценарии делегирования:
- read-heavy exploration по репозиторию и контрактам;
- проверка внешней документации и API-поведения;
- evidence collection, trace mapping, sample review;
- reviewer-pass перед фиксацией результатов фазы.

Ограничения:
- parallel write-heavy изменения не считать режимом по умолчанию;
- финальные архитектурные решения не делегировать без review родительского агента;
- subagent не утверждает завершённость фазы сам по себе;
- ответственность за интеграцию результатов и соответствие `master-spec.md` остаётся за родительским агентом.

Если subagents в окружении отсутствуют или не нужны, работа продолжается в single-agent режиме без изменения требований master-spec.

## Режим работы агента

### Обязательная последовательность на каждой задаче

0. Исследовать  
1. Реализовать  
2. Протестировать  
3. Отчитаться

Перед началом заметного шага агент должен отдельно оценить, улучшит ли качество результата явное привлечение subagent-а для read-heavy / evidence-heavy / review-heavy части работы. Если нет — продолжать single-agent исполнение.

### Что нужно возвращать после каждой законченной фазы

- перечень созданных/изменённых файлов;
- краткое описание реализованного слоя;
- evidence: какие тесты, sample payload и trace подтверждают выполнение;
- список оставшихся рисков и допущений;
- список `SPEC_GAP`, если они выявлены.

## Ограничения реализации

### n8n

Разрешено:

- стандартные ноды n8n;
- явные ветвления, conditions, HTTP Request, data mapping, sub-workflows;
- декомпозиция на несколько workflow ради читаемости.

Запрещено:

- `Code`, `Function`, `Function Item`;
- длинные expressions, превращающие workflow в спрятанный код;
- business logic в нечитабельных inline-преобразованиях;
- “временные” обходы, которые фактически становятся постоянной архитектурой.

### Сервисы

Допускаются только явно специфицированные вспомогательные сервисы:

- ingestion pipeline;
- при необходимости отдельно специфицированный retrieval API;
- адаптеры Lead Target API / CRM.

Нельзя самовольно вводить middleware-сервис, который перетянет на себя основную оркестрацию.

## Политика изменения спецификации

Если реализация упирается в неясность:

1. Не изобретать решение молча.
2. Зафиксировать `SPEC_GAP`.
3. Предложить минимальное уточнение.
4. Не менять архитектурные инварианты без новой версии master-spec.

## Минимальный baseline доказательств

До заявления о готовности baseline должны существовать:

- workflow-файлы фаз 1–10;
- sample payload для ключевых сценариев;
- acceptance matrix;
- traces по `correlation_id`;
- доказательство dedup / anti-loop;
- доказательство grounded RAG;
- доказательство идемпотентной записи лидов;
- отсутствие запрещённых нод в workflow.

## Definition of Failure

К дефектному исполнению также относится ситуация, когда subagents использованы так, что продуктовая архитектура фактически изменилась, а это никак не отражено в спецификации и evidence.


Работа считается дефектной, если выполнено хотя бы одно из условий:

- в workflow есть запрещённые code-ноды;
- логика спрятана в expressions;
- реализация перепрыгнула через dedup/anti-loop;
- CRM связана напрямую, минуя абстракцию Lead Target API;
- RAG отвечает выдумками при пустом или слабом retrieval;
- отсутствует evidence по завершённой фазе.
